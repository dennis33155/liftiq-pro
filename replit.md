# Workspace

## Overview

pnpm workspace monorepo using TypeScript. Each package manages its own dependencies.

## Artifacts

- **Gym Log** (`artifacts/gym-log`) — Native iOS app built with Expo + Expo Router. Most data persists on-device via AsyncStorage; AI features (workout suggestion, coach recommendations) call the api-server. Dark theme with crimson accent. Tabs: Train, History, Coach, Settings. Stack screens: workout (active session with rest timer + progression suggestion + per-exercise muscle chips + PR celebration modal that triggers when a completed set beats the current best by weight or reps; custom header has X cancel + small home icon that returns to Home tab without canceling the active workout), exercise-picker (modal, with "Recommended for you" carousel + muscle chips per row), custom-exercise (modal), exercise-detail/[id] (modal with Primary/Secondary muscle tabs + scrollable muscle list, last/suggest stats, preparation/execution/comment), workout-detail/[id] (with muscle chips per block), pr-history/[id] (per-exercise trend chart with metric toggle for est-1RM / top-set / volume + session log), body-metrics (weight + body-fat logging with AsyncStorage `gymlog.bodyMetrics.v1` and SVG trend chart toggleable between weight / body fat), progress-photos (camera + library picker, photos copied via expo-file-system to documentDirectory/progress-photos/, metadata in AsyncStorage `gymlog.progressPhotos.v1`, day-grouped 3-col grid with full-screen viewer and long-press delete). Home shows training streak badge (`lib/streak.ts` consecutive days ending today or yesterday). Coach tab links to body-metrics and progress-photos screens, and PR rows are tappable to open pr-history. Reusable `components/SimpleLineChart.tsx` (react-native-svg) drives the PR chart and body-metric chart. Celebration modal (`components/PRCelebrationModal.tsx`) uses Animated gold flash + auto-dismiss 4.5s + tap to dismiss. All non-tab stack screens have a `headerRight` (`components/HeaderRight.tsx`) that combines a small live time/date readout (`components/LiveDateTime.tsx`) with the Home icon (`components/HeaderHomeButton.tsx`) which calls `router.dismissAll()` then `router.replace("/")`. LiveDateTime also appears on every tab screen (Train, History, Coach, Settings) and ticks aligned to minute boundaries. Active workout header shows "Started <HH:MM> · now <HH:MM> · <day>"; workout-detail shows start time and end time alongside the date. Rest timer modal supports tap-to-dismiss backdrop (with `accessibilityLabel`) and accepts a `restartToken` so each set completion forcibly resets the countdown to 1:30 even if the modal is already visible. Progress photo persistence is cross-platform: on web (or for inline blob/data picker results) the image is converted via `fetch` + `FileReader.readAsDataURL` to a base64 data URI stored directly in AsyncStorage; on native it is copied into `documentDirectory/progress-photos/` via the modern `expo-file-system` `new File(..).copy()` API. `loadProgressPhotos` prunes stale entries (missing native files, dead `blob:` URLs) and `deleteProgressPhoto` skips the file delete for inline URIs. Categories: Chest, Back, Legs, Arms, Shoulders, Full Body. ~80 seeded exercises (including 11 user-specific machines: Hammer Strength Incline/Supine Press, Foreman FLY-231 Chest Fly, Dumbbell Pullover, Reverse Grip Lat Pulldown, Machine Lateral Raise, Dumbbell Shrugs, Foreman FLY-217 Low Row Rear Delt, Cable Curl, Cable Overhead Triceps Extension, DB Curl Arm Blaster, Hip Adductor) with primaryMuscles, secondaryMuscles, equipment, preparation, execution, tip metadata. Personal-note tips on key lifts (incline DB press priority, lat pulldown superset, shoulder press buildup, squat knee-sleeve note, MAG row grip, etc.). On first launch (or once after a flag bump), `lib/initialPRSeed.ts` auto-seeds 5 historical workouts (one per training day in the previous Wed→Sun training week, anchored to the most recent past Sunday at 18:30) containing the user's PR weights/reps as completed sets, with universal warm-up + Hip Abductor finisher; persisted-once via `gymlog.prSeed.v2` flag in storage. Seed workouts are merged into any existing history (sorted by startedAt desc) — never clobber. Bump the prSeed version key to re-deliver new seed content to existing devices. Unknown-weight exercises (Face Pull, Leg Curl, Calf Raises) seed with `weight: null` to avoid polluting Personal Bests with 0-lb records. components/MuscleChips renders primary muscles as filled crimson chips and secondary as outlined chips; reused across workout, picker, and workout-detail. lib/recommendation.ts ranks exercises by freshness and builds suggested workouts. lib/schedule.ts encodes the user's fixed weekly split (Wed Arms, Thu Back, Fri Shoulders [INJECTION DAY], Sat Chest [T PEAK], Sun Legs [T PEAK], Mon/Tue rest), plus universal warm-up (Triceps Dip 15 + Pull Up 12) and finisher (Hip Abductor) IDs. WorkoutContext seeds warm-up + finisher into every startWorkout/startSuggestedWorkout. Home shows TODAY card + 7-day strip + ALL CATEGORIES override section. AI Workout Suggestion: each CategoryCard and the TodayCard show a crimson "AI" pill (cpu icon); tapping opens a modal with optional notes (e.g. "low energy", "focus on chest peak") and a Generate button that calls `POST /api/workout-suggestion` (Claude sonnet-4-5, Zod-validated request/response, in-memory rate limit 10/min/IP with periodic eviction sweep). Server filters returned exerciseIds against availableExercises; client (WorkoutContext.startAiWorkout) dedupes IDs, requires >=2 valid exercises, wraps with universal warmup+finisher, sets active workout, and shows the rationale via Alert. Generation is guarded by a useRef in-flight lock to prevent double-submit races. AI Progress Photo Analysis: progress-photos screen has a "Select Photos to Analyze" mode (toggle button); tapping photos in select mode toggles a blue ring + checkmark; the "Analyze Photos" action POSTs each selected photo as a base64 data URI to `/api/analyze-progress-photo` (sequential loop, progress shown as N/M). Single-photo analysis is also available via the photo viewer modal ("Analyze This Photo" button). Server route uses OpenAI Responses API with `gpt-5-mini` vision (`detail: "low"`, `max_output_tokens: 400`) via the Replit AI Integrations OpenAI proxy (`AI_INTEGRATIONS_OPENAI_BASE_URL`/`_API_KEY`), in-memory rate limit 6/min/IP, validates payload with zod (data URI regex + 7MB cap). System prompt restricts output to fitness-focused topics only (muscle development, symmetry, posture, qualitative body comp, training focus); explicitly forbids medical advice, exact body fat numbers, shaming, sexual content, identity comments; falls back to a fixed safety string for non-physique photos. Body limit is path-scoped (`express.json({limit:"8mb"})` mounted ONLY on `/api/analyze-progress-photo`, before the global 64kb parser) so other routes stay protected from large-payload DoS. Analysis text (capped 800 chars) plus `analyzedAt` + `model` is stored on the photo as an optional `analysis` field in the existing AsyncStorage record (backward compatible). Thumbnails of analyzed photos show a small blue "AI" badge; the viewer modal renders the analysis in a blue tinted card. A persistent disclaimer card under the action buttons reads: "Photos are analyzed only when you tap Analyze Photos."

Coach tab (`app/(tabs)/coach.tsx`): shows a Personal Bests list (top 5 by estimated 1RM with Show all toggle) computed by `lib/personalRecords.ts` (Epley formula `weight * (1 + reps/30)` over completed sets in completed workouts; per-exercise: best 1RM/weight/reps, last weight/reps/date, total sessions); plus an AI Recommendations panel that POSTs `{notes?, personalRecords[], recentWorkouts[], availableExercises[]}` to `/api/coach` and persists the latest response under `gym-log:coach-recs:v1` in AsyncStorage. Recommendations card renders headline, focusExercises (chip + reason), progressionPlan (cards with sets x reps @ weight), neglectedAreas, weeklyTip; "Updated X min ago" timestamp. API URL resolved via `process.env.EXPO_PUBLIC_DOMAIN` in lib/api.ts. NativeTabs with isLiquidGlassAvailable() fallback.

- **API Server** (`artifacts/api-server`) — Express 5 API on the shared proxy under `/api`. Routes: `/healthz`, `POST /workout-suggestion`, `POST /coach`. Workout-suggestion route accepts `{ category, count?, notes?, recentWorkouts, availableExercises }`, asks Claude (claude-sonnet-4-5 via Replit AI Integrations env vars `AI_INTEGRATIONS_ANTHROPIC_BASE_URL` and `AI_INTEGRATIONS_ANTHROPIC_API_KEY`) for a JSON plan of `exercises[{exerciseId,sets,reps,note?}]` plus `rationale`, validates the response with Zod, filters returned IDs against `availableExercises`, returns the curated suggestion. Coach route accepts `{ notes?, personalRecords[{exerciseName,category,estimated1RM,bestWeight,bestReps,lastWeight,lastReps,lastDate,totalSessions}], recentWorkouts[{date,category,exerciseNames}], availableExercises[{name,category,primaryMuscles?}] }`, asks Claude to recommend ONLY exercises from the provided list, returns `{recommendations:{headline, focusExercises[], progressionPlan[], neglectedAreas[], weeklyTip}, createdAt}`. Both routes share the same in-memory token-bucket rate limiter pattern with periodic eviction (workout-suggestion 10/min/IP, coach 8/min/IP). Express JSON body limit is set globally to 25mb in app.ts. The body-analysis feature has been fully removed from both client and server.

## Stack

- **Monorepo tool**: pnpm workspaces
- **Node.js version**: 24
- **Package manager**: pnpm
- **TypeScript version**: 5.9
- **API framework**: Express 5
- **Database**: PostgreSQL + Drizzle ORM
- **Validation**: Zod (`zod/v4`), `drizzle-zod`
- **API codegen**: Orval (from OpenAPI spec)
- **Build**: esbuild (CJS bundle)

## Key Commands

- `pnpm run typecheck` — full typecheck across all packages
- `pnpm run build` — typecheck + build all packages
- `pnpm --filter @workspace/api-spec run codegen` — regenerate API hooks and Zod schemas from OpenAPI spec
- `pnpm --filter @workspace/db run push` — push DB schema changes (dev only)
- `pnpm --filter @workspace/api-server run dev` — run API server locally

See the `pnpm-workspace` skill for workspace structure, TypeScript setup, and package details.
